# 健康台灣深耕計畫｜內部工作會議系統

> 台北市醫師公會「健康台灣深耕計畫」的內部工作會議線上系統，整合提案收集、工作回報、列管事項追蹤，並可一鍵產生符合公文格式的會議議程 Word 檔。純前端、零後端、免安裝。

---

## 專案簡介

這是一套提供台北市醫師公會「健康台灣深耕計畫－臺北市慢性病防治全人健康智慧整合照護計畫」團隊使用的內部工作會議系統。整個系統由三個獨立的 HTML 頁面組成，彼此透過共用的 Firebase Realtime Database 串接資料，從「會前蒐集提案與工作回報」到「會中產出正式議程文件」形成完整流程：

1. **提案收集平台**（`proposal.html`）— 與會人員線上送出提案與工作回報，管理員審閱。
2. **議程製作系統**（`agenda.html`）— 確認歷次會議列管事項辦理情形，套入提案與處方／拓點統計，產出議程 Word 檔。
3. **入口頁**（`index.html`）— 系統首頁與導覽。

所有資料儲存於 Firebase 雲端，無需自行架設伺服器或資料庫。

---

## 主要功能

### 入口頁（`index.html`）
- 系統首頁，提供主題化的卡片式導覽，連結至各子系統。

### 提案收集平台（`proposal.html`）
供與會人員與管理員使用，採頁籤式介面（送出提案／工作回報／我的紀錄／管理端）：

- **送出提案**
  - 可手動填寫「提案人、案由、說明、建議放置位置（討論事項／報告事項／臨時動議）」。
  - 也可上傳提案單檔案（PDF 或 Word `.docx`），透過 AI 自動解析並填入欄位，確認後再送出。
  - 支援由管理員設定的「與會人員名單」快速選取自己的名字。
- **工作回報**
  - 與會人員填寫「近期工作執行情形（本週）」與「未來工作重點規劃（下週）」，供議程系統自動統整為工作進度表。
- **我的紀錄**
  - 依姓名檢視自己送出過的提案及其審閱狀態。
- **管理端**（需輸入 PIN 碼登入，預設 `0000`）
  - 檢視所有提案，依狀態（待審閱／已納入／不納入／需修改）與事項類別篩選。
  - 標記提案狀態、刪除提案。
  - 檢視所有人的工作回報彙整表。
  - 設定「提案截止時間」：提案頁顯示倒數，截止後自動鎖住收件。
  - 管理「與會人員名單」：新增／移除人員，供提案與工作回報頁快速選取。

> 提案經管理員標記為「已納入」後，議程製作系統可直接載入。

### 議程製作系統（`agenda.html`）
供承辦人員製作會議議程，核心為「歷次決議列管事項」的維護與議程文件產出：

- **列管事項確認**
  - 逐筆填入列管事項（來源會次、案由、決議事項摘要、會議日期）。
  - 每筆可選擇「繼續列管」或「解除列管」。
  - 「本次辦理情形」可選「依決議辦理」或「另外說明」；於「另外說明」輸入口語描述後，按 **AI 生成** 可產出正式公文格式的辦理情形，預覽確認後採用。
  - 支援新增「空白摘要筆記」。
  - 內容變更時自動儲存至 Firebase 雲端（debounce 自動存檔）。
- **議程資料來源整合**（透過 Firebase 共用資料）
  - 自動載入提案平台中「已納入」的提案，依報告／討論／臨時動議分區。
  - 自動載入工作回報，產生「工作進度表」。
- **Excel 統計自動運算**
  - 上傳處方執行 Excel，依本週／上週時間窗計算「運動／營養／社會／情緒調適處方」的開立數、執行數與執行率，產生處方統計表。
  - 整理診所拓點現況表（分類、教學時間、電子簽章、營養空間、禮券、開立／執行數等）。
- **產出 Word 文件**
  - 一鍵產出完整會議議程 `.docx`（含主席致詞、計畫推動現況、歷次決議辦理情形、報告／討論事項、臨時動議等章節），採標楷體與公文版面設定。
  - 亦可單獨產出「歷次決議辦理情形.docx」。
- **歷次存檔**
  - 將議程記錄存檔至雲端，可於「歷次存檔」清單中重新下載。

> 部分 AI 功能（提案解析、辦理情形生成）透過外部 Gemini 代理服務 `https://meetbot-check-in-system.onrender.com/gemini-proxy` 進行。

## 技術棧

| 項目 | 說明 |
|------|------|
| 前端框架 | React 18（UMD 版，透過 CDN 載入） |
| JSX 編譯 | Babel Standalone（瀏覽器端即時編譯，`type="text/babel"`） |
| 資料儲存 | Firebase Realtime Database（前端直連，無自建後端） |
| Excel 解析 | [SheetJS / xlsx](https://sheetjs.com)（CDN） |
| Word 讀取 | [Mammoth.js](https://github.com/mwilliamson/mammoth.js)（讀取 `.docx` 純文字，CDN） |
| Word 產生 | **`docx.js`**（[docx](https://docx.js.org) 函式庫的瀏覽器打包版，約 742 KB，置於 repo 內供前端離線載入；用於在瀏覽器端產生 `.docx` 議程檔） |
| AI 服務 | 外部 Gemini 代理 API（提案解析、辦理情形文字生成） |
| 樣式 | 內嵌 CSS（無 CSS 框架），字體 Noto Serif TC |

**特性：**

- 純前端靜態網站，**無建置步驟（no build）**、無 `package.json`、無打包工具。
- 所有相依套件除 `docx.js` 外皆由 CDN 載入；JSX 於瀏覽器端由 Babel 即時編譯。
- 無自建後端伺服器；資料持久化全數仰賴 Firebase。

> `docx.js` 為第三方函式庫，僅作為前端產生 Word 檔之用，毋需閱讀或修改其內容。

## 如何使用

本系統為純靜態網站，無需安裝任何相依套件，但因使用了 ES 模組（Firebase）與檔案讀取，**建議透過靜態伺服器開啟**，以避免瀏覽器對 `file://` 協定的限制。

### 方式一：靜態伺服器（建議）

於專案根目錄執行任一靜態伺服器，再以瀏覽器開啟對應網址：

```bash
# Python 3
python -m http.server 8000

# 或 Node.js（npx，免安裝）
npx serve
```

接著於瀏覽器前往 `http://localhost:8000/index.html`。

### 方式二：直接開啟 HTML

亦可直接以瀏覽器開啟 `index.html`。若遇到 Firebase 模組或檔案上傳相關功能無法運作，請改用方式一。

### 使用流程

1. **會前**：與會人員開啟 `proposal.html`，於「送出提案」頁送出提案，並於「工作回報」頁填寫工作進度。
2. **審閱**：管理員以 PIN 碼登入 `proposal.html` 管理端，審閱提案並標記為「已納入」，視需要設定提案截止時間與維護人員名單。
3. **製作議程**：承辦人員開啟 `agenda.html`，確認歷次列管事項辦理情形、上傳處方 Excel 統計，系統會自動帶入已納入提案與工作回報，最後一鍵下載議程 `.docx`。

> 本系統限內部使用，所有資料儲存於 Firebase 雲端。
